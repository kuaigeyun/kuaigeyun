# RiverEdge SaaS 多组织框架 - 架构设计文档

## 📋 项目概述

**RiverEdge SaaS 多组织框架 (RiverEdge SaaS Multi-tenant Framework)** 是一套基于插件化架构的 SaaS 多组织管理框架，采用前后端分离设计，支持动态加载应用插件。

## 🏗️ 设计哲学

**RiverEdge SaaS 多组织框架**采用**主流 B 端项目设计**，遵循企业级应用的最佳实践：

- **infra** (基础设施): 平台级后端，提供平台运营和管理功能
- **core** (核心): 系统级后端，提供组织内系统管理功能
- **server** (服务入口): 应用入口层，整合平台级和系统级功能
- **apps** (应用插件): 应用级功能模块，可扩展的业务功能模块
- **routes** (路由): 前端路由配置，定义应用路由规则
- **components** (组件): 前端公共组件，可复用的 UI 组件
- **services** (服务): 前端 API 服务层，封装后端 API 调用

所有命名都遵循主流 B 端项目规范，清晰明确，易于理解和维护。

## 🏗️ 架构设计

### FIR 技术栈 ⭐ **核心技术组合**

**RiverEdge** 采用 **FIR** 技术栈组合，构建现代化企业级 SaaS 应用：

- **F** - **FastAPI**: 高性能异步 Web 框架，提供 RESTful API 服务
- **I** - **Inngest**: 现代化任务调度系统，处理异步任务和业务流程
- **R** - **React**: 现代化前端框架，构建响应式用户界面

**FIR 技术栈优势**：
- ✅ **高性能**: FastAPI 提供异步高并发能力，Inngest 处理复杂异步任务
- ✅ **现代化**: React 18+ 提供最佳用户体验，Inngest 提供可靠的任务调度
- ✅ **可扩展**: FastAPI 插件化架构，Inngest 支持复杂工作流，React 组件化开发
- ✅ **开发效率**: FastAPI 自动生成 API 文档，Inngest 可视化工作流，React 生态丰富
- ✅ **企业级**: 完整的错误处理、日志、监控、部署方案

### 框架三层结构 ⭐ **核心架构概念**

RiverEdge SaaS 多组织框架采用**三层结构设计**，清晰划分不同层级的职责和功能。不同层级采用不同的架构设计方法：

- **平台级、系统级**：采用**传统分层架构设计**，设计较为固定，功能稳定，便于维护
- **应用级**：采用**DDD（领域驱动设计）架构设计**，支持复杂业务逻辑和领域建模

#### 1. 平台级（Platform Level）

**定位**：平台整体运营和管理
- **目标用户**：超级管理员
- **架构设计**：**传统分层架构设计**
  - 采用经典的三层架构：Controller → Service → Repository
  - 设计较为固定，功能稳定，便于维护
  - 适合平台运营、监控、配置管理等标准化功能
- **核心职责**：
  - 平台整体运营数据统计
  - 所有组织的统一管理
  - 平台系统监控和运维
  - 套餐配置和管理
  - 平台级配置和策略

**典型功能**：
- 运营看板（平台整体数据）
- 组织管理（所有组织的统一管理）
- 系统状态监控（平台系统健康状态）
- 套餐管理（平台套餐配置）

**访问路径**：`/platform/*`
- `/platform/operation` - 运营看板
- `/platform/tenants` - 组织管理
- `/platform/monitoring` - 系统状态
- `/platform/packages` - 套餐管理

**权限要求**：仅超级管理员可见

#### 2. 系统级（System Level）

**定位**：当前组织的系统管理
- **目标用户**：组织管理员、普通用户
- **架构设计**：**传统分层架构设计**
  - 采用经典的三层架构：Controller → Service → Repository
  - 设计较为固定，功能稳定，便于维护
  - 适合用户管理、角色权限、组织配置等标准化功能
- **核心职责**：
  - 当前组织的数据统计
  - 组织内用户和角色管理
  - 组织配置管理
  - 系统级功能设置

**典型功能**：
- 仪表盘（当前组织的数据统计）
- 用户管理（组织内用户管理）
- 角色管理（组织内角色和权限管理）

**访问路径**：`/dashboard`, `/users`, `/roles`

**注意**：租户管理功能属于平台级功能，由超级管理员在 `/platform/tenants` 进行管理。系统级不包含租户管理功能，组织管理员如需查看组织信息，应通过平台级组织管理功能。

**权限要求**：组织管理员或普通用户可见（根据权限不同显示不同功能）

#### 3. 应用级（Application Level）

**定位**：业务应用功能
- **目标用户**：业务用户
- **架构设计**：**DDD（领域驱动设计）架构设计**
  - 采用领域驱动设计（DDD）架构
  - 支持复杂业务逻辑和领域建模
  - 通过插件系统实现，每个业务应用独立设计
  - 适合 MES、ERP、WMS、CRM 等复杂业务场景
- **核心职责**：
  - 业务应用的数据展示
  - 业务功能实现
  - 业务流程管理
  - 领域模型设计和实现

**典型功能**：
- MES 生产管理（生产计划、工单管理、质量检验等）
- ERP 企业资源管理（采购、库存、财务等）
- CRM 客户关系管理（客户管理、销售管理、服务管理等）
- 其他业务应用

**访问路径**：通过插件系统动态加载
- 插件路由：`/plugins/{plugin-name}/*`

**权限要求**：根据业务应用权限控制

**架构特点**：
- **领域模型**：通过 DDD 架构设计领域模型，清晰表达业务概念
- **业务逻辑封装**：业务逻辑封装在领域层，避免贫血模型
- **插件化设计**：每个业务应用作为独立插件，可动态加载和卸载
- **领域服务**：通过领域服务实现复杂的业务规则和流程
- **聚合根**：使用聚合根管理领域对象的生命周期和一致性
- **领域事件**：通过领域事件处理业务流程和状态转换
- **值对象**：使用值对象表达业务概念，保证不变性

#### 三层结构对比

| 层级 | 定位 | 目标用户 | 架构设计 | 核心职责 | 典型功能 | 访问路径 |
|------|------|----------|----------|----------|----------|----------|
| **平台级** | 平台整体运营 | 超级管理员 | **传统分层架构** | 平台运营和管理 | 运营看板、组织管理、系统监控 | `/platform/*` |
| **系统级** | 组织系统管理 | 组织管理员、普通用户 | **传统分层架构** | 组织内系统管理 | 仪表盘、用户管理、角色管理 | `/dashboard`, `/users`, `/roles`（不包含租户管理） |
| **应用级** | 业务应用功能 | 业务用户 | **DDD 架构** | 业务功能实现 | MES、ERP、WMS、CRM 等 | `/plugins/*` |

#### 三层结构设计优势

1. **职责清晰**：每层职责明确，避免功能混淆
   - 平台级专注于平台整体运营，不涉及具体业务
   - 系统级专注于组织内系统管理，不涉及平台运营
   - 应用级专注于业务功能实现，不涉及系统管理

2. **架构适配**：不同层级采用适合的架构设计方法
   - 平台级、系统级采用传统分层架构，设计固定，便于维护
   - 应用级采用 DDD 架构，支持复杂业务逻辑和领域建模
   - 架构选择与业务复杂度匹配，提升开发效率和代码质量

3. **权限隔离**：不同层级有不同的权限要求，保证安全性
   - 平台级：仅超级管理员可见，保证平台安全
   - 系统级：组织内权限控制，保证数据隔离
   - 应用级：业务权限控制，保证功能安全

4. **可扩展性**：应用级通过插件系统动态扩展，不影响核心系统
   - 平台级和系统级功能稳定，不频繁变更
   - 应用级功能通过插件动态加载，灵活扩展
   - 新业务应用不影响现有系统
   - DDD 架构支持领域模型的演进和扩展

5. **易于维护**：层级划分清晰，便于代码组织和维护
   - 代码结构清晰，职责明确
   - 修改某一层级不影响其他层级
   - 便于团队协作和代码审查
   - 传统分层架构简单直观，DDD 架构支持复杂业务建模

6. **用户体验**：不同角色看到不同的功能，界面更简洁
   - 超级管理员看到平台级功能
   - 组织管理员看到系统级功能
   - 业务用户看到应用级功能
   - 避免功能冗余，提升用户体验

7. **数据隔离**：不同层级的数据访问范围不同
   - 平台级：可访问所有组织数据（跨组织）
   - 系统级：仅访问当前组织数据（组织隔离）
   - 应用级：访问业务数据（业务隔离）

#### 三层结构实现方式

**平台级实现（代码目录：infra/）**：
- **后端代码目录**：`src/infra/`（基础设施层模块）
  - 代码模块名使用 `infra`，避免与 Python 关键字冲突
  - 采用传统分层架构：Controller → Service → Repository
- **后端 API 路由**：`/api/v1/infra/*`（对应 infra/ 文件夹）
  - 跨组织数据访问（`skip_tenant_filter=True`）
  - 平台级数据统计和管理
  - 仅超级管理员权限验证
  - 示例：`/api/v1/infra/tenants`、`/api/v1/infra/packages`、`/api/v1/infra/monitoring`
- **前端路由**：`/platform/*`（用户访问路径）
  - 运营看板、组织管理、系统监控等
  - 仅超级管理员可见
- **菜单显示**：平台管理菜单（仅超级管理员可见）
- **数据范围**：所有组织的数据（跨组织访问）

**说明**：代码目录使用 `infra/`（基础设施层），API 路径使用 `/api/v1/infra/*`（对应后端文件夹），前端路由保持 `/platform/*`（用户访问路径）。

**系统级实现**：
- **后端 API**：`/api/v1/core/*` 路由（对应 core/ 文件夹，组织隔离）
  - 自动组织过滤（`tenant_id` 隔离）
  - 组织内数据管理
  - 组织管理员和普通用户权限验证
  - 示例：`/api/v1/core/users`、`/api/v1/core/roles`、`/api/v1/core/permissions`
- **前端路由**：`/dashboard`, `/users`, `/roles` 等路由
  - 仪表盘、用户管理、角色管理等
  - 根据用户权限显示不同功能
- **菜单显示**：系统管理菜单（根据用户权限显示）
- **数据范围**：当前组织的数据（组织隔离）

**应用级实现**：
- **后端代码目录**：`src/apps/`（应用级功能模块）
  - 采用 DDD（领域驱动设计）架构
  - 支持复杂业务逻辑和领域建模
  - 每个应用插件独立设计和实现
- **后端 API 路由**：`/api/v1/apps/{插件名}/*` 或 `/api/v1/{插件名}/*`（通过插件系统注册）
  - 插件定义自己的 API 路由
  - 通过 FastAPI Router 注册到核心系统
  - 业务权限验证
  - 示例：`/api/v1/apps/kuaimes/orders`、`/api/v1/apps/kuaicrm/customers`
- **前端代码目录**：`src/apps/`（应用级前端模块）
  - 每个应用插件包含独立的页面、服务、类型定义
- **前端路由**：插件路由（通过插件系统动态加载）
  - 插件定义自己的前端路由
  - 通过 React Router 动态注册
  - 根据插件权限显示
- **菜单显示**：插件菜单（根据插件权限显示）
- **数据范围**：业务数据（业务隔离，自动组织隔离）

#### 三层结构数据访问规则

**平台级数据访问**：
- ✅ 可以访问所有组织的数据
- ✅ 可以跨组织统计和分析
- ✅ 可以管理所有组织
- ❌ 不访问具体业务数据

**系统级数据访问**：
- ✅ 只能访问当前组织的数据
- ✅ 自动进行组织隔离（`tenant_id` 过滤）
- ✅ 可以管理组织内用户、角色等
- ❌ 不能访问其他组织的数据
- ❌ 不访问具体业务数据

**应用级数据访问**：
- ✅ 只能访问当前组织的业务数据
- ✅ 自动进行组织隔离
- ✅ 可以访问业务相关的所有数据
- ❌ 不能访问其他组织的数据
- ❌ 不能访问系统级管理数据

#### 三层结构权限控制

**平台级权限控制**：
- 权限检查：`is_superuser=True`
- 访问控制：仅超级管理员可以访问
- 数据范围：所有组织数据

**系统级权限控制**：
- 权限检查：`is_tenant_admin` 或普通用户权限
- 访问控制：根据用户角色显示不同功能
- 数据范围：当前组织数据（自动隔离）

**应用级权限控制**：
- 权限检查：业务应用定义的权限
- 访问控制：根据业务权限显示功能
- 数据范围：当前组织的业务数据（自动隔离）

### 1. riveredge-backend (后端核心系统) - 基础设施层 + 系统级

**职责：**
- **基础设施层功能（infra）**：
  - 平台整体运营数据统计
  - 所有组织的统一管理 API
  - 平台系统监控 API
  - 套餐配置管理 API
- **系统级功能**：
  - 提供核心 API 服务
  - 用户认证与授权
  - 权限管理
  - 系统配置管理
  - 组织内数据管理
- **基础设施**：
  - 插件注册与管理
  - 数据持久化
  - 多组织数据隔离

**技术栈（FIR 后端部分）：**
- **编程语言**: Python 3.11 LTS (长期支持版本，推荐3.11.9)
- **Web 框架**: FastAPI 0.115.0+ ⭐ **FIR 核心** (主流稳定版，高性能异步 Web 框架)
- **ASGI 服务器**: Uvicorn 0.32.0+ (主流稳定版，FastAPI 官方推荐)
- **任务调度**: Inngest >=0.3.0 ⭐ **FIR 核心** (现代化任务调度系统，处理异步任务和工作流)
- **数据验证**: Pydantic 2.9.0+ (主流稳定版，与FastAPI 0.115+完美兼容)
- **数据库**: PostgreSQL 15+ (推荐15.6+，主流稳定版)
- **ORM 框架**: Tortoise ORM 0.21.1 (主流稳定版，与FastAPI 0.115+完美兼容)
- **数据库迁移**: Aerich 0.7.1+ (主流稳定版，Tortoise ORM 数据库迁移工具)
- **缓存**: Redis 6.2.6+ (推荐6.2.6+，BSD 3-Clause 许可证，完全免费商业使用)
- **认证**: JWT (python-jose[cryptography] 3.3.0+)
- **API 文档**: FastAPI 自动生成 OpenAPI/Swagger
- **HTTP 客户端**: httpx 0.27.0+
- **日志**: loguru 0.7.3+
- **配置管理**: pydantic-settings 2.6.0+
- **测试框架**: pytest 8.0.0+ + pytest-asyncio 0.23.0+

详细技术选型请参考：[1.技术选型.md](./1.技术选型.md)

### 后端插件机制

#### 插件注册方式

**模块注册模式**（采用）
- 插件作为 Python 模块注册到核心系统
- 路由由 FastAPI 统一管理和路由
- 共享认证、权限、数据库等基础设施
- 使用 Python 动态导入 (`importlib`) 加载插件
- **Inngest 集成**: 插件可以定义 Inngest 函数，处理异步任务和工作流 ⭐ FIR 核心

#### 后端插件功能

- **API 路由注册**: 插件通过 FastAPI Router 注册自己的 RESTful API 路由
- **Inngest 任务**: 插件可以定义 Inngest 函数，处理异步任务和工作流 ⭐ FIR 核心
- **权限管理**: 插件定义所需的权限，由核心系统统一验证（JWT）
- **数据模型**: 插件使用 Tortoise ORM 定义自己的数据实体和数据库表
- **数据库迁移**: 插件使用 Aerich 管理数据库迁移脚本
- **业务逻辑**: 插件实现独立的业务逻辑和服务
- **Pydantic Schema**: 插件使用 Pydantic v2 定义数据验证模型

#### 后端插件实现

**插件基类**:
```python
from abc import ABC, abstractmethod
from fastapi import APIRouter

class BasePlugin(ABC):
    """插件基类"""
    
    @abstractmethod
    def get_router(self) -> APIRouter:
        """返回插件的 API 路由"""
        pass
    
    @abstractmethod
    def get_models(self) -> list:
        """返回插件的 Tortoise ORM 模型列表"""
        pass
```

**应用插件加载流程**:
1. 扫描 `src/apps/` 目录下的所有应用插件
2. 动态导入插件的 Python 模块
3. 通过 FastAPI Router 注册 API 路由
4. 注册插件的数据库模型（Tortoise ORM）
5. 执行插件的数据库迁移（Aerich）
6. 注册插件的 Inngest 函数（如果存在）⭐ FIR 核心

详细实现方案请参考：[插件开发规范.md](../4.branch/插件开发规范.md)

### 前端插件机制

### 功能热插拔机制 ⭐

#### 设计理念

功能热插拔是指在不修改核心代码的情况下，通过插件机制动态加载和卸载功能模块（如支付、短信、邮件等），实现功能的灵活扩展和替换。

**核心原则**：
- **核心代码零污染**：核心系统只定义接口，不包含任何具体实现
- **接口标准化**：通过抽象接口定义功能契约，插件实现具体功能
- **运行时热插拔**：支持在不重启服务的情况下启用/禁用功能
- **多实现支持**：同一功能可以有多个实现（如多种支付方式）

#### 架构设计

```
riveredge-backend/
├── src/
│   ├── infra/
│   │   └── domain/
│   │       └── interfaces/      # 功能接口定义（核心）
│   │           ├── payment.py   # 支付接口
│   │           ├── sms.py       # 短信接口
│   │           ├── email.py     # 邮件接口
│   │           └── storage.py   # 存储接口
│   └── core/
│       └── service_registry.py  # 服务注册表（核心）
│   └── apps/                    # 应用插件（可选，功能热插拔实现）
│       ├── payment-alipay/      # 支付宝支付插件
│       ├── payment-wechat/      # 微信支付插件
│       ├── sms-aliyun/          # 阿里云短信插件
│       ├── sms-tencent/         # 腾讯云短信插件
│       └── email-smtp/          # SMTP 邮件插件
```

#### 实现流程

1. **接口定义**：核心系统定义功能接口（如 `IPaymentProvider`、`ISMSProvider`）
2. **服务注册表**：核心系统提供服务注册表，管理所有服务提供商
3. **插件实现**：插件实现接口，在 `on_load()` 钩子中注册到服务注册表
4. **核心调用**：核心系统通过服务注册表获取服务提供商并调用

#### 典型应用场景

- **支付功能**：支付宝、微信支付、银联等
- **短信服务**：阿里云、腾讯云、华为云等
- **邮件服务**：SMTP、SendGrid、阿里云邮件等
- **存储服务**：OSS、COS、S3 等
- **消息推送**：极光推送、个推、Firebase 等
